## Codebase Patterns

### ProviderType shared schema
- **Where**: `packages/workflows/src/schemas/provider.ts`
- **Pattern**: Provider enum extracted to its own file to avoid circular imports between workflow.ts and dag-node.ts. Re-exported from schemas/index.ts and core/types/index.ts.
- **Example**: `import type { ProviderType } from './schemas/provider';`

### Config type widening pattern
- **Where**: `packages/core/src/config/config-types.ts`, `packages/workflows/src/deps.ts`
- **Pattern**: All `'claude' | 'codex'` union types replaced with `ProviderType`. New provider defaults interfaces added. WorkflowConfig.assistants must include all 4 providers as required keys.
- **Example**: `assistant: ProviderType` instead of `assistant: 'claude' | 'codex'`

### Tool implementation signature
- **Where**: `packages/core/src/clients/tools/*.ts`
- **Pattern**: All tools export `async function xxxTool(params: Record<string, unknown>, cwd: string): Promise<string>`. Path validation via shared `path-validation.ts` using `validatePath(filePath, cwd)`.
- **Example**: `import { readTool } from './tools/read';`

### MCP client injectable spawn pattern
- **Where**: `packages/core/src/clients/mcp-client.ts`
- **Pattern**: `McpToolProvider` accepts `spawnFn` option for testing since `spyOn` on `require`'d `child_process.spawn` doesn't intercept static imports. Constructor injection avoids mock.module() pollution.
- **Example**: `new McpToolProvider(configs, { spawnFn: fakeSpawn })`

### Tool loop patterns
- **Where**: `packages/core/src/clients/tool-loop.ts`
- **Pattern**: `executeToolLoop(config)` is an `AsyncGenerator<MessageChunk>`. Uses a `toolRegistry` map for dispatch. SSE parsing via `parseSSEStream()`. Tool call deltas assembled via `assembleToolCalls()`.
- **Key types**: `ToolLoopConfig`, `ChatMessage`, `ToolCall`, `ChatCompletionChunk`, `ChatCompletionResponse`, `ProviderEndpointConfig`
- **Feature params**: `allowedTools`, `deniedTools`, `systemPrompt`, `outputFormat`, `outputFormatStyle` are processed at the start of `executeToolLoop()` — callers pass these via config, not pre-processing.
- **Hooks**: `ToolLoopHooks` with `PreToolUse`, `PostToolUse`, `PostToolUseFailure` matchers. `ToolLoopHookMatcher` has `matcher?` (regex), `hooks[]` (callbacks), `timeout?`. Hook payloads match Claude SDK shapes. `runHooks()` evaluates all matching hooks; errors propagated (not swallowed).
- **MCP integration**: `mcpProvider` field on `ToolLoopConfig`. MCP tools merged into tool list. MCP tool calls (prefixed `mcp__`) dispatched to `McpToolProvider.callTool()`. Caller manages lifecycle (connect/shutdown).
- **Skills integration**: `skillContext` field on `ToolLoopConfig`. System prompt additions appended after user messages. Skill `toolAllowlist` intersected with `allowedTools` when both are set.

### Output format style pattern
- **Where**: `packages/core/src/clients/openai-compatible.ts`, `packages/core/src/clients/llamacpp.ts`
- **Pattern**: `getOutputFormatStyle()` protected hook returns `'response_format'` (default) or `'grammar'` (Llama.cpp). Passed to tool loop's `outputFormatStyle` config field. The tool loop handles the actual mapping.
- **Example**: `protected override getOutputFormatStyle(): 'response_format' | 'grammar' { return 'grammar'; }`

### Codex dual-path dispatch pattern
- **Where**: `packages/core/src/clients/codex.ts`
- **Pattern**: `needsToolLoopFallback(options)` checks if any unsupported features are requested. If yes, `sendQueryViaToolLoop()` uses the OpenAI API directly with the Codex model. If no, the native SDK path runs.
- **Unsupported features**: tools, disallowedTools, hooks, mcpConfigs, skills, systemPrompt, effort, thinking, maxBudgetUsd, fallbackModel, betas, sandbox
- **Natively supported**: model, modelReasoningEffort, webSearchMode, additionalDirectories, outputFormat (as outputSchema), abortSignal

### Codex test batch isolation
- **Where**: `packages/core/package.json`
- **Pattern**: `codex.test.ts` runs in its own `bun test` batch because it `mock.module()`s `./tool-loop`, `./mcp-client`, `./skill-loader`, `./context-window`, `./tool-definitions` which would pollute those modules' own tests.
- **Example**: `bun test src/clients/claude.test.ts && bun test src/clients/codex.test.ts && bun test src/clients/context-window.test.ts ...`

### Pre-existing test failures
- `packages/core/src/orchestrator/prompt-builder-knowledge.test.ts`: "truncates logs that exceed token budget" — fails on main branch
- `packages/cli/src/commands/workflow-run.test.ts`: 6 workflowRunCommand tests fail on main branch
